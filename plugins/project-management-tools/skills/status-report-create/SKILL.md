# Status Report Create

## Shared Reference

Use `../../references/project-management-principles.md` for general project
management principles. Apply status reporting, risk, stakeholder communication,
and change control sections.

## Reporting Stance

Write status as a decision tool. A good status report shows what changed, what
is on track or off track, what needs attention, and what help or decisions are
needed.

## Workflow

1. Identify reporting audience, period, project objective, and milestone plan.
2. Summarize progress, accomplishments, upcoming work, risks, issues, changes,
   and decisions needed.
3. Assign RAG status with a short evidence-based explanation.
4. Separate FYI updates from sponsor asks.
5. Flag unclear or missing data rather than inventing status.

## Output

Return:

- Executive summary.
- RAG status: green, amber, or red with rationale.
- Progress against milestones.
- Key accomplishments.
- Upcoming work.
- Risks and issues.
- Decisions needed.
- Scope, schedule, cost, or quality changes.
- Help requested.
- Next update date or cadence.

## Common Failure Modes

- Reporting activity instead of progress.
- Green status with unmanaged risks.
- No clear ask for sponsor decisions.
- Hiding scope or schedule changes.
- Omitting blockers because owners are unclear.

## Quality Bar

The report should let a sponsor quickly understand project health and decide
whether intervention is needed.
