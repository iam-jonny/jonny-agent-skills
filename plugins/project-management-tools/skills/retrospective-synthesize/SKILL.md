# Retrospective Synthesize

## Shared Reference

Use `../../references/project-management-principles.md` for general project
management principles. Apply retrospective, risk, planning, stakeholder, and
common failure mode sections.

## Retro Stance

Focus on learning that changes future behavior. Separate facts, observations,
root causes, decisions, and improvement actions. Avoid vague lessons that no
one can implement.

## Workflow

1. Identify project or milestone context, participants, and review scope.
2. Extract what went well, what did not, surprises, root causes, and patterns.
3. Separate one-off events from repeatable system issues.
4. Convert lessons into action items, owners, due dates, and process changes.
5. Recommend what should be added to templates, checklists, governance,
   communication, risk management, or planning.

## Output

Return:

- Retrospective summary.
- What went well.
- What did not go well.
- Root causes and patterns.
- Decisions or principles to keep.
- Action items with owners and due dates.
- Process, template, or governance changes.
- Follow-up check date.
- Risks if improvements are not implemented.

## Common Failure Modes

- Producing observations without owners.
- Blaming individuals instead of improving systems.
- No root cause analysis.
- Actions are too broad to verify.
- Lessons do not feed into future planning or risk management.

## Quality Bar

The retro output should create specific behavior changes that can be checked
later.
